# SUVIDHA Deployment Guide

## Frontend (Vercel/Netlify)

We support two ways to deploy the client on Vercel:

### Option A: Root-Level Deployment (Automatic)
Simply link the root directory of this repository to Vercel.
The root-level `vercel.json` and `.vercelignore` will automatically handle the build and routing:
- **Framework Preset:** `Vite` (automatically detected or set to `Other`)
- **Install Command:** `npm install --prefix client` (managed by `vercel.json`)
- **Build Command:** `npm run build --prefix client` (managed by `vercel.json`)
- **Output Directory:** `client/dist` (managed by `vercel.json`)

### Option B: Directory-Specific Deployment (Manual)
If you prefer, you can set the **Root Directory** to `client` in the Vercel project settings:
- **Framework preset:** `Vite`
- **Install command:** `npm install`
- **Build command:** `npm run build`
- **Output directory:** `dist`
- **Routing:** `client/vercel.json` provides the SPA fallback routing to `index.html`.

### Required Environment Variables
Configure the following in the Vercel Dashboard under project settings:
- `VITE_API_URL=https://your-backend-api-domain.com` (exclude trailing slash)
- `VITE_KIOSK_ID=kiosk-01`
- `VITE_KIOSK_KEY=<device-key>`
- `VITE_APP_VERSION=1.0.0`


## Backend (Render/Railway/AWS)

We support two ways to deploy the backend on Render:

### Option A: Render Blueprint Deployment (Recommended & Automatic)
We have configured a `render.yaml` Blueprint file at the root. This will automatically spin up both a PostgreSQL database and the Node.js Express service:
1. Go to your Render Dashboard and click **New > Blueprint**.
2. Connect your Git repository.
3. Render will auto-detect the `render.yaml` file.
4. It will automatically:
   - Provision a PostgreSQL database (`suvidha-db`).
   - Link the database connection string (`DATABASE_URL`) to the Web Service.
   - Generate secure, random values for `JWT_SECRET` and `HMAC_SECRET`.
5. You will only need to input:
   - `CORS_ORIGIN`: Set this to your frontend URL (e.g. `https://your-frontend.vercel.app`).
   - `APP_BASE_URL`: Set this to the public URL of the backend Web Service generated by Render (e.g., `https://suvidha-backend.onrender.com`).
6. Click **Apply** to start the deployment.

### Option B: Manual Web Service Deployment
If you prefer manual setup (or if using the Free Tier where pre-deploy commands are restricted):
- **Service Type:** Web Service
- **Root Directory:** `server`
- **Language:** `Node`
- **Build Command:** `npm install && npx prisma generate`
- **Start Command:** `npx prisma migrate deploy && npm run seed:admin && npm start`
- **Environment Variables:** Provide all variables listed in the required environment variables section below, and link a PostgreSQL database.


### Required Environment Variables
If setting up manually or using the blueprint, ensure these are configured:
- `DATABASE_URL`: Connection string for PostgreSQL database.
- `JWT_SECRET`: Secure secret key for JWT verification.
- `HMAC_SECRET`: Secure secret key for HMAC validation.
- `CORS_ORIGIN`: Comma-separated list of allowed origins (e.g., Vercel frontend URL).
- `APP_BASE_URL`: The public URL of the backend service (used for generating absolute upload links).
- `KIOSK_DEVICE_KEYS`: Device authentication keys formatted as `kiosk-01:key1,kiosk-02:key2`.
- `JWT_EXPIRES_IN`: Optional, defaults to `7d`.
- Optional Twilio/SMTP credentials for SMS/Email notifications.


## Database (Neon/Supabase/RDS)

- Local development default uses SQLite (`DATABASE_URL=file:./dev.db`).
- For production, set a PostgreSQL `DATABASE_URL` (Neon/Supabase/RDS).

- Create PostgreSQL database
- Set `DATABASE_URL` in backend env
- Run migration:

```bash
npx prisma migrate deploy
```

## Production Checklist

- Set strong secrets for JWT/HMAC
- Restrict CORS to frontend domain
- Enforce HTTPS at platform level
- Configure persistent object storage (S3/Cloudinary) for uploads
- Configure SMS and email providers
- Set monitoring and log retention

## Container Deployment

- Build and run locally:

```bash
docker compose up --build
```

- Client: `http://localhost:8080`
- API Gateway: `http://localhost:5000`
- Core service: `http://localhost:5003`
- Auth service: `http://localhost:5101`
- Payment-fraud service: `http://localhost:5102`

Gateway routing contracts:

- `/api/auth/*` → auth-service
- `/api/payments/*` → payment-fraud-service
- `/api/services|complaints|documents|admin|sync/*` → core service

## Kubernetes Deployment Baseline

Manifests are available in `k8s/`:

- `server-deployment.yaml` (API deployment + service)
- `auth-service-deployment.yaml`
- `payment-fraud-service-deployment.yaml`
- `notification-service-deployment.yaml`
- `api-gateway-deployment.yaml`
- `client-deployment.yaml` (frontend deployment + service)
- `ingress.yaml` (TLS ingress and path routing)
- `hpa.yaml` (horizontal autoscaling)
- `mtls-istio.yaml` (service-to-service STRICT mTLS policy)

Apply in sequence:

```bash
kubectl apply -f k8s/server-deployment.yaml
kubectl apply -f k8s/auth-service-deployment.yaml
kubectl apply -f k8s/payment-fraud-service-deployment.yaml
kubectl apply -f k8s/notification-service-deployment.yaml
kubectl apply -f k8s/api-gateway-deployment.yaml
kubectl apply -f k8s/client-deployment.yaml
kubectl apply -f k8s/ingress.yaml
kubectl apply -f k8s/hpa.yaml
kubectl apply -f k8s/mtls-istio.yaml
```
